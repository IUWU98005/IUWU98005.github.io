GitFlow 是什么?#

GitFlow 简单来说, 就是使用 git 进行版本控制的一套规范.

根据 GitFlow 的建议, 主要的分支分为 master, hotfix, dev, release 和 feat 这 5 种分支.

下图来自 why-need-git-flow:

  • master : 主干分支, 用于正式发布的分支, 需要由专人严格审核测试后的 release / hotfix 分支才可以合并到该分支.

  • hotfix : 用于热修复, 当 master 出现紧急 bug 时, 就需要基于 master 临时创建一个分支进行修复, 修复测试完成后, 合并到 master 和 dev 分支, 修复完成后删除 hotfix 分支.

  • dev : 开发分支, 在 master 分支基础上创建而来, 是功能开发的基础分支. 一般不可在这条分支上直接开发, 需要新建一个 feat 分支, 再进行开发.

  • feat: 功能分支, 基于 master 分支, 不同的开发人员负责不同的 feat 分支, 开发测试完成后, 提交 PR(Pull Request) 到 remote, 由专人审核, 审核通过后合并到 dev 分支.

  • release: 预发布分支, 当 dev 分支积累了一定的功能后, 会基于 dev 新建一个 release 分支, 用于完成正式发布前的准备工作, 通过测试后, 最终会合并回 master 和 dev 分支

git 相较于其他版本管理工具的优缺点#

优点#

  1. 分布式架构

    每个开发者都有完整的仓库副本(含历史记录) ,不依赖中央服务器即可提交和查看历史.

    优势：离线操作、高可用性、更快的本地操作

  2. 分支管理强大

    分支轻量、快速,支持多人并行开发、CI/CD流程.

    合并(merge/rebase) 工具成熟,利于团队协作.

    Git Flow / trunk-based development 等都以 Git 为基础.

  3. 速度快

    大多数操作(如 diff、log、commit) 都是在本地进行,不需要联网访问远程服务器.

    对于大项目也具有优秀的性能表现.

  4. 数据完整性高

    每个提交通过 SHA-1 哈希校验,防篡改能力强.

    历史记录不可随意修改(除非有意如 rebase) .

缺点#

  1. 不适合处理大二进制文件

    Git 本质是为代码服务的文本型系统,大文件(如图片、音视频、模型等) 管理困难.

    虽然有 Git LFS(Large File Storage) 方案,但配置和使用略复杂.

  2. 仓库膨胀风险

    因为每个用户本地都有完整仓库副本,历史记录多时体积变大,clone 较慢.

  3. 操作自由,导致误操作风险高

    如 force push、rebase 不当使用可能覆盖历史.

    SVN 中则较少发生这种情况(更受控) .

git 提交时, 如果出现冲突该如何解决#

  1. 首先使用 git pull 命令, 将 remote 的代码拉取到本地.
  2. 编译器一般会弹出窗口提示用户手动解决冲突 merge
  3. 在解决冲突的时候, 手动编辑需要保留的最终版本
  4. 解决完所有冲突并commit后, 再将本地代码 push 到 remote 上
  5. 在 gitlab 或者 github 上提交 pull request, 请求将 feat 分支合并到 dev
